編譯器是將高階程式語言（如 C、C++、Java）撰寫的原始碼，翻譯成特定機器架構上之組合語言或機械碼的複雜系統軟體。與只進行簡單文字替換或一對一翻譯的組譯器不同，編譯器必須理解高階語言的複雜語意、控制流與資料結構，並進行最佳化以提升執行效率。

本章將帶領讀者拆解編譯器的內部運作，理解從字串到機械指令的六大編譯階段。

---

## 5.1 編譯器的基本結構

現代編譯器為了提升跨平台移植性與程式碼最佳化的效率，通常會被劃分為**前端 (Frontend)** 與**後端 (Backend)** 兩大區塊，兩者之間透過**中間表示法 (Intermediate Representation, IR)** 進行溝通。

```
原始碼 ===> [ 前端 (Frontend) ] ===> 中間表示碼 (IR) ===> [ 後端 (Backend) ] ===> 目標碼
```

### 5.1.1 編譯的前端與後端
- **前端 (Frontend)**：與**來源語言**相關。負責分析原始碼的語法與語意，確認程式符合語言規範，並建立中間代碼 (IR)。前端不需要關心程式最終運行在哪個 CPU 上。
- **後端 (Backend)**：與**目標機器**相關。負責對中間代碼進行最佳化，將其翻譯成特定 CPU 的機器指令，並管理暫存器的分配。後端不需要關心程式最初是用什麼高階語言寫成的。

### 5.1.2 編譯的六個主要階段
在前端與後端內部，編譯過程被進一步細分為六個連續的階段：

1. **詞法分析 (Lexical Analysis)**：讀取原始碼字元流，將其組合並識別為有意義的標記（Tokens）。
2. **語法分析 (Syntactic Analysis)**：確認 Token 的排列順序是否符合語言的文法結構，並建立語法樹 (Parse Tree / AST)。
3. **語意分析 (Semantic Analysis)**：檢查語意正確性，例如變數是否先宣告後使用、型態是否相容（Type Checking）。
4. **中間碼產生 (Intermediate Code Generation)**：將語法樹轉換為平台無關的低階指令序列（如三位址碼）。
5. **程式碼最佳化 (Code Optimization)**：對中間碼進行邏輯調整，以提高運行速度或減少記憶體佔用。
6. **目標碼產生 (Code Generation)**：將最佳化後的中間碼轉譯為目標機器的機械碼或組合語言。

## 5.2 詞法分析 (Lexical Analysis / Scanning)

詞法分析器（又稱 Scanner）是編譯器的第一關，它的任務是過濾掉空白、換行與註解，並把原始碼的字元流切分成一個個的**記號 (Token)**。

### 5.2.1 記號 (Token) 的定義與分類
Token 是程式中具備獨立語意的最小單元。常見的 Token 分類包括：
- **關鍵字 (Keywords)**：`if`, `else`, `while`, `int`, `return`
- **識別字 (Identifiers)**：變數名、函式名（如 `x`, `counter`, `main`）
- **常數 (Constants)**：整數 `42`、浮點數 `3.14`、字串 `"Hello"`
- **運算子 (Operators)**：`+`, `-`, `*`, `/`, `&&`, `==`
- **分界符 (Delimiters)**：`;`, `,`, `(`, `)`, `{`, `}`

### 5.2.2 正規表示式與有限狀態自動機 (FSA)
為了解析 Token，詞法分析器使用數學工具來定義 Token 的規則：
1. **正規表示式 (Regular Expressions, Regex)**：用來精確定義 Token 的文字格式。例如，定義識別字規則為「字母開頭，後面接任意個字母或數字」：
   $$\text{Identifier} = [a-zA-Z][a-zA-Z0-9]*$$
2. **有限狀態自動機 (Finite State Automata, FSA)**：編譯器會把正規表示式轉換成狀態機圖形。程式在讀取字元時，會隨之在各個「狀態」之間轉移。當讀到非法字元或進入「接受狀態」時，即成功切出一個 Token。
   - **確定有限自動機 (DFA)**：對於任一輸入字元，狀態轉移都是唯一的，這是詞法分析器在程式碼實作時最常用的形式。

## 5.3 語法分析 (Syntactic Analysis / Parsing)

語法分析器（Parser）接收 Token 流，並判定其排列是否符合該語言的語法規則。

### 5.3.1 上下文無關文法 (Context-Free Grammar, CFG)
高階語言的語法通常使用**上下文無關文法**與 **巴科斯範式 (BNF)** 來定義。例如，定義一個簡單的算術表達式：
$$E \to E + T \mid T$$
$$T \to T * F \mid F$$
$$F \to \text{id} \mid ( E )$$
利用文法規則，Parser 會將 Token 流（如 `x + y * z`）組合成一棵**抽象語法樹 (Abstract Syntax Tree, AST)**，樹的結構反映了運算子的優先順序（先乘除後加減）。

### 5.3.2 語法分析的主要演算法
語法分析主要分為兩大派系：

#### 1. 由上而下分析法 (Top-Down Parsing)
從文法的起始符號開始，嘗試推導出與 Token 流相符的結構。
- **遞迴下降分析 (Recursive Descent Parsing)**：最直觀的寫法。為文法中的每個非終端符號撰寫一個遞迴函式。當遇到多個分支時，透過讀取下一個 Token (Lookahead) 來決定進入哪個分支。
- **LL(1) 分析**：使用一個顯式的解析堆疊與 LL(1) 解析表來預測推導步驟。「L」代表從左至右讀取，「L」代表最左推導，而「1」代表每次只看 1 個 Token。

#### 2. 由下而上分析法 (Bottom-Up Parsing)
從 Token 流出發，藉由將 Token「歸約 (Reduce)」回文法符號，逐步向上構建出語法樹的根節點。
- **移位-歸約分析 (Shift-Reduce Parsing)**：使用堆疊來儲存符號。當堆疊頂端的符號符合某條文法規則的右側時，就將其「歸約」為左側的符號（例如將 `F` 歸約為 `T`）。
- **LR 分析**：功能最強大的語法分析方法，能處理絕大多數程式語言的語法。常見的變體包括 SLR、LALR 與 canonical LR。現代許多編譯器工具（如 Yacc/Bison）就是基於 LALR 演算法自動產生 Parser 程式碼的。

## 5.4 中間碼產生與最佳化

語法和語意分析通過後，編譯器會生成平台無關的**中間表示法 (IR)**。

### 5.4.1 三位址碼 (Three-Address Code)
三位址碼（3AC）是一種最常見的 IR 格式，其中每條指令最多隻能有三個記憶體位址（兩個運算元，一個結果）。例如，高階語言的 `x = a + b * c` 會被拆解為：
```assembly
t1 = b * c
t2 = a + t1
x = t2
```
在編譯器內部，三位址碼通常以下列資料結構表示：
- **四元組 (Quadruples)**：包含 `(op, arg1, arg2, result)`。例如 `(*, b, c, t1)`。這種格式易於進行程式碼最佳化。
- **三元組 (Triples)**：省去暫存變數 `t1`，改用指令的編號/指標來參照其運算結果。

### 5.4.2 程式碼最佳化 (Code Optimization)
最佳化旨在使產生的目標碼執行更快或佔用更少空間。常見的獨立於機器的最佳化技術包括：
- **常數摺疊 (Constant Folding)**：在編譯期就將常數運算算出。例如將 `x = 3 + 5` 直接替換為 `x = 8`。
- **死碼刪除 (Dead Code Elimination)**：移除永遠不會執行到的程式碼（如 `if (false) { ... }`）或宣告了卻從未使用的變數。
- **共同子運算式消除 (Common Subexpression Elimination)**：如果程式中多次重複計算相同的表達式，則只計算一次並保留結果。例如：
  ```c
  // 最佳化前
  x = a + b;
  y = (a + b) * c;
  
  // 最佳化後
  x = a + b;
  y = x * c;
  ```

## 5.5 目標碼產生 (Code Generation)

編譯器的最後階段是將最佳化後的中間碼，翻譯成特定目標 CPU（如 x86、ARM 或 SIC/XE）的機械指令。這個階段有兩個核心難題：

### 5.5.1 暫存器分配 (Register Allocation)
CPU 內部的高速暫存器數量非常有限，但中間碼中可能使用了成百上千個臨時變數（如 `t1, t2, ...`）。
- **暫存器配置任務**：決定在哪個時間點，把哪些變數放入實體暫存器中，哪些變數則必須「溢出 (Spill)」放回速度較慢的主記憶體堆疊區。
- **圖形著色演算法 (Graph Coloring)**：現代編譯器常用的一種高度優化的暫存器分配演算法。它將變數的生命週期重疊關係建立為圖形，並透過圖形染色問題來決定暫存器的指派。

### 5.5.2 指令選擇 (Instruction Selection)
相同的運算在不同的 CPU 上可能有多種指令組合。例如，將變數加 1 可以使用 `ADD` 指令，也可以使用更快速的 `INC` 指令。目標碼產生器必須智慧地選擇最優的指令序列，並配合定址模式（如相對定址），輸出最終的組合語言程式碼或二進位執行檔。
