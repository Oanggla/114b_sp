在系統程式的開發與維護過程中，除了核心的翻譯與連結載入程式外，我們每天都在使用的工具還包括**文字編輯器 (Text Editors)** 與**除錯系統 (Debugging Systems)**。

作為系統軟體的一部分，文字編輯器必須能夠高效地在記憶體中維護與編輯超大型文字資料；而除錯器（如 GDB）則需要深入作業系統與硬體底層，對執行中的行程進行監視與控制。本章將介紹這兩大工具的內部設計架構與運作原理。

## 7.1 文字編輯器的設計與結構

文字編輯器看似簡單，但如果只是將整個檔案讀入一個簡單的字元陣列中，當檔案大小達到數百 MB 時，任何一次局部的字元插入或刪除都會導致後面所有字元的記憶體搬移，造成極為嚴重的卡頓。因此，編輯器核心需要特殊的資料結構來管理記憶體中的文字。

### 7.1.1 編輯器核心：文字緩衝區管理 (Text Buffer Management)
現代編輯器常用以下三種資料結構來管理文字緩衝區：

#### 1. 行指標陣列 (Array of Line Pointers)
- **結構**：將檔案依「換行字元 (`\n`)」切分成多行，每行是一個獨立的動態字元陣列。編輯器使用一個指標陣列來儲存每一行的記憶體位址。
- **優點**：容易取得任意指定行，且修改某一行時不會影響到其他行。
- **缺點**：如果整份檔案只有極長的一行，此方法將退化成簡單陣列；且頻繁插入/刪除行會導致指標陣列本身的重配與搬移。

#### 2. 間距緩衝區 (Gap Buffer)
- **結構**：在當前游標（Cursor）所在的位置，預留一段未使用的空白記憶體空間（稱為 Gap）。
- **運作機制**：
  - 當使用者輸入字元時，直接寫入 Gap 的起始處，並縮小 Gap。
  - 當游標移動時，編輯器會把游標移動路徑上的字元搬移到 Gap 的另一側，使 Gap 永遠維持在游標當前位置。
- **優點**：在游標附近進行連續插入與刪除非常快速，幾乎不需要記憶體複製。
- **缺點**：當游標在大檔案中進行跨度極大的移動時，需要搬移大量字元以搬遷 Gap，開銷較大。
- **典型應用**：Emacs。

```
[游標前文字] | H | e | l | l | o |   |   |   |   |   | [游標後文字] w | o | r | l | d |
                                 ^               ^
                                 | <--- Gap ---> |
```

#### 3. 繩狀結構 (Rope)
- **結構**：一種基於**二元樹 (Binary Tree)** 的資料結構。樹的葉子節點（Leaf Nodes）儲存實際的短字串片段，內部節點（Internal Nodes）則儲存其子樹所含的總字元長度。
- **運作機制**：
  - 插入操作：將某個葉子節點分裂，插入新節點，並重新平衡二元樹。
  - 刪除操作：直接移除特定節點或調整範圍。
- **優點**：支援極大檔案的任意位置快速插入、刪除與分割，複雜度僅為對數時間 $O(\log N)$。非常利於實現**復原 (Undo) / 重做 (Redo)** 機制（只需保留舊的子樹指標）。
- **缺點**：遍歷與連續讀取速度不如連續記憶體。

### 7.1.2 使用者介面對應與編輯命令處理
文字編輯器包含一個命令處理迴圈（Command Loop）：
1. 監聽鍵盤與滑鼠事件。
2. 將實體輸入對應至邏輯編輯命令（例如：按下 Backspace 對應「刪除游標前字元」命令）。
3. 修改底層的 Text Buffer 資料結構。
4. **顯示映射 (Display Mapping)**：計算 Buffer 修改後，視窗畫面上哪些行需要被重新繪製（Redraw），並呼叫圖形或終端機 API 更新畫面。

## 7.2 除錯系統 (Debugging Systems) 的基本功能與設計

除錯器（Debugger）是程式開發者排查錯誤的核心武器。一個合格的除錯器必須具備：單步執行（Step Over/Into）、變數觀測、記憶體讀寫、以及設定**斷點 (Breakpoints)** 的功能。

### 7.2.1 符號式除錯 (Symbolic Debugger) 的運作機制
當我們在原始碼中看到 `print x` 時，CPU 其實根本不知道什麼是 `x`。CPU 只知道暫存器或特定的記憶體位址（如 `[EBP - 8]`）。
* **偵錯資訊格式 (Debug Information Format)**：在編譯程式時，若加上偵錯旗標（如 GCC 的 `-g`），編譯器會在可執行檔中額外寫入偵錯符號表，現代 Linux 系統最常使用 **DWARF 格式**。
* **符號映射**：DWARF 記錄了變數名稱與記憶體位址的映射、以及機器碼指令位址與原始碼行號的對應關係。除錯器正是讀取了這些資訊，才能在使用者輸入變數名時，正確去記憶體中抓取數值呈現在畫面上。

### 7.2.2 斷點 (Breakpoints) 的實現原理
斷點是除錯器中最重要的功能。根據實現方式，斷點分為兩類：

#### 1. 軟體斷點 (Software Breakpoints)
軟體斷點的數量沒有限制，其運作方式非常巧妙，包含「偷樑換柱」的步驟：
1. **指令備份**：當使用者在位址 `Addr` 設定斷點時，除錯器會備份該位址上的**原創機器指令**（保留 1 位元組）。
2. **指令替換**：除錯器將該位址的第一個位元組，替換為特殊的「中斷觸發指令」。在 x86 架構下為 `INT 3`（機器碼為 `0xCC`，長度剛好為 1 位元組）；在其他架構下則是特定的陷阱指令 (Trap)。
3. **觸發中斷**：當被除錯程式執行到 `Addr` 時，CPU 執行到 `INT 3`，會自動觸發硬體異常，並暫停程式，將控制權交回作業系統。
4. **訊號捕捉**：作業系統向除錯器發送異常訊號（如 `SIGTRAP`）。除錯器攔截此訊號，更新 UI 顯示程式已在該行暫停。
5. **恢復執行**：當使用者指示繼續執行時，除錯器會暫時將備份的原創指令寫回該位址，單步執行過去後，再重新寫回 `0xCC`。

#### 2. 硬體斷點 (Hardware Breakpoints)
- **原理**：利用 CPU 晶片內部提供的**偵錯暫存器 (Debug Registers)**（例如 x86 中的 `DR0` 到 `DR7`）。
- **運作**：除錯器將要監控的記憶體位址寫入這些暫存器中。每當 CPU 存取（讀、寫或執行）這些位址時，硬體會自動觸發中斷。
- **優缺點**：不需要修改記憶體中的程式碼，且能監控「資料寫入/讀取」（資料斷點 Watchpoints）。但受限於硬體暫存器數量，通常只能設定 4 個左右的硬體斷點。

### 7.2.3 系統呼叫追蹤與除錯控制介面
除錯器能夠控制另一個行程，是因為作業系統核心提供了專用的系統呼叫。
在 Linux 系統中，這個核心 API 是 **`ptrace` 系統呼叫**。
- **`ptrace` 功能**：允許一個行程（除錯器）監視並控制另一個行程（被除錯程式）的執行，並能直接讀取和寫入該行程的暫存器與虛擬記憶體空間。
- 除錯器透過 `ptrace(PTRACE_SINGLESTEP, ...)` 指令，可以命令 CPU 只執行一條指令就暫停，從而實現精準的「單步執行」。
