在組合語言程式設計中，程式設計師經常需要重複編寫某些相似的指令序列。為了減少重複的打字工作並降低出錯率，系統軟體提供了**巨集 (Macro)** 功能。

巨集處理器的核心任務是進行**文字替代 (Text Substitution)**：它將原始碼中定義的巨集名稱，展開為其預先定義好的完整指令序列。本章將介紹巨集的基礎概念、一行程巨集處理器的設計與資料結構，以及各種進階巨集功能。

---

## 3.1 巨集定義與巨集擴充

### 3.1.1 巨集指令與子程式的比較
初學者常將「巨集 (Macro)」與「子程式 (Subroutine/Function)」混淆。雖然兩者都能達到程式碼重複使用的目的，但它們的實現層面和執行期行為有本質上的不同：

| 比較維度 | 巨集 (Macro) | 子程式 (Subroutine) |
| :--- | :--- | :--- |
| **處理時期** | **組譯前/編譯前**（由巨集處理器處理）。 | **執行期**（由 CPU 透過跳躍指令執行）。 |
| **運作機制** | **文字擴充**。程式碼中每次呼叫巨集，其定義的指令序列就會被原封不動地複製到該位置。 | **控制權轉移**。利用 `JSUB` / `RSUB` 跳躍到子程式位址，執行完後返回。 |
| **記憶體開銷** | **高**。呼叫次數愈多，產生的機械碼檔案就愈大（程式碼膨脹）。 | **低**。不論呼叫多少次，子程式在記憶體中只有一份實體。 |
| **執行速度** | **快**。沒有執行期呼叫子程式（保護暫存器、跳躍與返回）的額外開銷 (Overhead)。 | **慢**。每次呼叫都有執行期控制權轉換與參數傳遞的開銷。 |

### 3.1.2 巨集處理器的基本功能
巨集處理器的運作可以分為兩個主要步驟：
1. **巨集定義 (Macro Definition)**：識別並儲存原始碼中定義的巨集。在組合語言中，通常使用 `MACRO` 與 `MEND` 虛擬指令來包夾定義。例如：
   ```assembly
   RDBUFF  MACRO   &INDEV, &BUFADR, &RECLEN
           LDX     #0
           TD      &INDEV
           ...
           MEND
   ```
2. **巨集擴充 (Macro Expansion)**：當在程式中偵測到已定義的巨集名稱（即「巨集呼叫 Macro Invocation」），巨集處理器會將其替換為對應的指令序列，並將其中的**形式參數 (Formal Parameters)** 替換為呼叫時傳入的**實際參數 (Actual Parameters)**。

## 3.2 一行程巨集處理器 (One-Pass Macro Processor) 設計

現代系統軟體中，巨集處理常作為組譯器的第一個前導步驟。為了提高處理效率，大多數巨集處理器被設計為 **一行程 (One-Pass)**。

### 3.2.1 為什麼巨集定義與擴充必須在同一個行程內處理？
在一個巨集定義內部，可能又呼叫了另一個巨集。為了支援**先定義後呼叫**以及**巢狀巨集 (Nested Macros)** 的處理，巨集處理器必須在讀取原始碼的同時，一邊儲存新定義的巨集，一邊將遇到呼叫的巨集展開。因此，單次掃描（One-Pass）是最適當的設計。

### 3.2.2 核心資料結構設計
一行程巨集處理器仰賴三張核心資料表來管理巨集定義與展開：

#### 1. DEFTAB (Definition Table，定義表)
用於儲存所有巨集的完整指令序列。
- 為了簡化參數替換，巨集中的形式參數（如 `&INDEV`）在存入 `DEFTAB` 時會被轉換成**位置參數標記**（如 `?1`, `?2`）。這樣在展開時，只需用 `ARGTAB` 中對應索引的值進行直接替換，而不需要進行字串搜尋。

#### 2. NAMTAB (Name Table，命名表)
用於記錄已定義的巨集名稱，並作為 `DEFTAB` 的索引。
- 欄位包含：巨集名稱、該巨集在 `DEFTAB` 中的起始指標（Start）、以及結束指標（End）。
- `NAMTAB` 通常使用雜湊表（Hash Table）以加速巨集呼叫時的名稱比對。

#### 3. ARGTAB (Argument Table，參數表)
在巨集擴充（展開）期間使用，用於暫存執行呼叫時傳入的實際參數值。
- 當遇到巨集呼叫時，傳入的實際參數（如 `F1`, `BUFFER`, `LENGTH`）會被按順序填入 `ARGTAB`。
- 展開程式碼時，巨集處理器會把 `DEFTAB` 中讀到的位置標記 `?n` 替換為 `ARGTAB[n]` 的字串。

```
[巨集呼叫] RDBUFF F1, BUFFER, LENGTH
                 |     |       |
               +---+--------+--------+
ARGTAB:        | 1 | BUFFER | LENGTH |   (實際參數表)
               +---+--------+--------+
                 |     |       |
                 v     v       v
[DEFTAB 內容]  TD  ?1  ==> TD  F1
               RD  ?2  ==> RD  BUFFER
```

### 3.2.3 巨集擴充演算法與處理流程
一行程巨集處理器在任何時間點都處於兩種狀態之一：
- **`EXPANDING = FALSE` (正常處理模式)**：讀取原始輸入檔。
  - 若讀到 `MACRO`：切換至定義模式，將巨集定義寫入 `DEFTAB` 與 `NAMTAB`。
  - 若讀到已在 `NAMTAB` 中的名稱：將實際參數寫入 `ARGTAB`，將狀態設為 `EXPANDING = TRUE`。
  - 其他指令：直接輸出至組譯器。
- **`EXPANDING = TRUE` (巨集展開模式)**：從 `DEFTAB` 讀取指令，而非輸入檔。
  - 讀取 `DEFTAB` 中的下一行指令，將位置參數 `?n` 替換為 `ARGTAB[n]`。
  - 若讀到 `MEND`：說明此巨集展開結束。如果沒有巢狀呼叫，則將狀態設回 `EXPANDING = FALSE`，繼續讀取原始輸入檔。
  - 輸出替換後的指令至組譯器。

## 3.3 進階與特殊的巨集功能

### 3.3.1 巨集定義中的巨集定義 (Concatenated Macro Definitions)
巨集處理器允許在巨集展開的過程中動態定義新的巨集。
* 例如：巨集 `MACA` 在被呼叫並展開時，其展開的內容包含了 `MACB MACRO ... MEND` 的定義敘述。這樣一來，只有在 `MACA` 被至少呼叫一次後，`MACB` 才會被納入 `NAMTAB` 與 `DEFTAB`，之後才能被程式呼叫。

### 3.3.2 唯一標記的自動生成 (Generation of Unique Labels)
如果巨集定義內部包含了分支跳躍指令，則需要用到標記（Labels）。
* **問題**：若同一個巨集在程式中被呼叫了多次，展開後會產生重複的標記定義（Duplicate Labels），導致組譯錯誤。
* **解決方案**：巨集處理器提供唯一標記生成機制。在定義中使用 `$` 開頭的標記（如 `$LOOP`）。每次展開該巨集時，巨集處理器會自動將 `$` 替換為唯一的序列字元（如 `$AA`, `$AB`, `$AC`），確保展開後的每一組標記都是獨一無二的。

### 3.3.3 條件式巨集擴充 (Conditional Macro Expansion)
條件式擴充允許根據巨集呼叫時的參數值或狀態，動態決定要展開哪些指令。這極大增加了組合語言的彈性。
- **`IF`, `ELSE`, `ENDIF`**：根據條件判斷，只將符合條件的指令段輸出展開。
- **`WHILE`, `ENDW`**：利用循環機制，重複展開某段指令。這在需要動態生成大量相似表格或重複初始化資料時非常有用。

### 3.3.4 關鍵字參數 (Keyword Parameters)
在基本巨集中，參數是按照位置對齊的（Positional Parameters）。當參數數量很多時，程式設計師很容易記錯順序。
* **關鍵字參數**允許在呼叫時直接指定參數名稱，且不限順序。例如：
  ```assembly
  RDBUFF  BUFADR=BUFFER, INDEV=F1, RECLEN=LENGTH
  ```
* 組譯器在定義巨集時，也可以為關鍵字參數指定預設值（Default Values）：
  ```assembly
  RDBUFF  MACRO   &INDEV=F1, &BUFADR=, &RECLEN=100
  ```
  如果呼叫時未傳入 `RECLEN`，則自動使用預設值 `100`。
