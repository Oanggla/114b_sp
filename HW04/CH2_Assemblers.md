組譯器是系統軟體中最基礎的翻譯程式。它的核心任務是將人類易讀的**組合語言 (Assembly Language)** 程式，翻譯成機器能直接執行的**機器語言 (Machine Language / Object Code)**。本章將以第一章介紹的 SIC/XE 架構為基礎，探討組譯器的工作原理、演算法設計與關鍵資料結構。

---

## 2.1 組譯器的基本功能與指令翻譯

組譯器的基本功能是將組合語言源碼中的每一行指令轉換成對應的機器碼，並產生可被載入執行的目的檔。

### 2.1.1 機器指令翻譯的核心步驟
在翻譯過程中，組譯器主要進行以下工作：
1. **操作碼翻譯**：將組合語言的助憶碼（Mnemonic，如 `LDA`, `ADD`）轉換成其等價的二進位操作碼（Opcode，如 `00`, `18`）。
2. **符號位址解析**：將程式中定義的標記（Labels/Symbols，如 `LOOP`, `RETADR`）轉換成記憶體中的實體或相對位址。
3. **資料常數生成**：將原始碼中定義的字元或整數常數（如 `C'EOF'` 或 `X'F1'`) 轉換成其對應的二進位表示。

為了完成這些工作，組譯器需要兩張核心資料表：
- **OPTAB (Operation Code Table，操作碼表)**：用於查詢助憶碼對應的機械碼、指令格式與長度。通常在組譯器設計時即已固定（靜態表）。
- **SYMTAB (Symbol Table，符號表)**：用於記錄程式中所有標記（Labels）及其對應的記憶體位址（Location）。此表是在組譯過程中動態建立的（動態表）。

### 2.1.2 組譯器虛擬指令 (Assembler Directives)
虛擬指令是給**組譯器本身**的指示，它們不會被翻譯成 CPU 執行的機器指令，但會影響記憶體配置或組譯行為。

在 SIC/XE 中，最常用的虛擬指令包括：
- **`START`**：指定程式的名稱與起始記憶體位址。例如：`COPY START 1000`。
- **`END`**：標示程式的結束，並可選擇性指定程式的起始執行點。例如：`END FIRST`。
- **`BYTE`**：宣告字元或十六進位常數，並佔用相應位元組。例如：`EOF BYTE C'EOF'`（佔 3 Bytes）或 `ZERO BYTE X'00'`（佔 1 Byte）。
- **`WORD`**：宣告一個 24 位元（1 字組）的整數常數。例如：`THREE WORD 3`（佔 3 Bytes）。
- **`RESB` (Reserve Bytes)**：保留指定數量的位元組空間，不初始化。例如：`BUFFER RESB 4096`。
- **`RESW` (Reserve Words)**：保留指定數量的字組空間。例如：`LENGTH RESW 1`（佔 3 Bytes）。

## 2.2 二行程組譯器 (Two-Pass Assembler) 設計

在組合語言中，我們經常會在定義某個符號之前就先使用它（例如跳躍指令 `J LOOP` 出現在標記 `LOOP` 的定義之前）。這被稱為**前向參考 (Forward Reference)**。
為了解決這個問題，標準的組譯器通常設計為 **二行程 (Two-Pass)** 結構。

### 2.2.1 Pass 1 演算法：位址分配與符號建立
Pass 1 的主要目的是掃描整份原始碼，計算每一行指令的記憶體位址，並建立符號表 (SYMTAB)。

#### Pass 1 核心邏輯 (虛擬碼)：
1. 讀取第一行（應為 `START` 指令）。
2. 初始化位置計數器 **`LOCCTR` (Location Counter)** 為 `START` 所指定的起始位址（若無指定則設為 0）。
3. 重複讀取下一行，直到讀入 `END`：
   - 如果該行有標記（Label）：
     - 檢查 `SYMTAB` 是否已存在此標記。若存在，表示重複定義錯誤。
     - 若不存在，將 `(Label, LOCCTR)` 寫入 `SYMTAB`。
   - 查詢 `OPTAB` 確認該指令是否為有效指令：
     - 若為有效指令，根據其指令格式（Format 1, 2, 3 或 4），將 `LOCCTR` 加上對應的長度（1, 2, 3 或 4 Bytes）。
     - 若是虛擬指令：
       - `WORD`：`LOCCTR += 3`
       - `RESW`：`LOCCTR += 3 * 運算元數值`
       - `BYTE`：`LOCCTR += 常數所佔位元組數`
       - `RESB`：`LOCCTR += 運算元數值`
4. 將最終的程式長度計算出來：$\text{Length} = \text{LOCCTR} - \text{Start Address}$。

### 2.2.2 Pass 2 演算法：程式碼生成與目的檔輸出
Pass 2 的主要目的是利用 Pass 1 建立好的 `SYMTAB`，將指令翻譯成機械碼，並輸出成結構化的目的檔。

#### Pass 2 核心邏輯 (虛擬碼)：
1. 讀取第一行（`START`）。
2. 初始化讀取狀態，寫入目的檔的 **Header Record**。
3. 重複讀取每一行，直到 `END`：
   - 若該行為機器指令：
     - 查詢 `OPTAB` 取得 Opcode。
     - 查詢 `SYMTAB` 取得運算元符號的位址（若是立即定址常數則直接使用其數值）。
     - 根據定址旗標（n, i, x, b, p, e）計算目標位址 (TA) 與位移量 (disp)，組合成機器碼。
   - 若為 `BYTE` 或 `WORD`：將常數值翻譯成對應的十六進位機器碼。
   - 將生成的機器碼暫存至緩衝區，當緩衝區滿或遇到記憶體不連續（如 `RESB`）時，將其寫入目的檔的 **Text Record**。
   - 若是 Format 4 指令且涉及外部符號或需要重定位，產生 **Modification Record**。
4. 寫入 **End Record**，標示翻譯完成。

### 2.2.3 目的檔格式 (Object Program Records)
組譯器輸出的目的檔是一系列文字格式的紀錄：

- **H (Header)**：記錄程式名稱、起始位址與程式總長度。
  ```
  H ^ 程式名稱(6字元) ^ 起始位址(6位數) ^ 程式長度(6位數)
  ```
- **T (Text)**：記錄要載入記憶體的實際機器碼。
  ```
  T ^ 起始載入位址(6位數) ^ 長度(2位數，表示此Record的Byte數) ^ 機器碼(十六進位)
  ```
- **M (Modification)**：記錄在載入期需要被重新定位的記憶體欄位（通常針對 Format 4 的絕對定址）。
  ```
  M ^ 需要修改的起始半位元組位址(6位數) ^ 修改長度(2位數，半位元組數)
  ```
- **E (End)**：標示目的檔結束，並指出程式開始執行的進入點位址。
  ```
  E ^ 進入點位址(6位數)
  ```

## 2.3 機器相關的組譯器功能

在 SIC/XE 架構下，組譯器必須處理多種指令格式與定址模式，特別是相對定址與重定位。

### 2.3.1 相對定址 (Relative Addressing) 的處理
在 Format 3 中，指令只有 12 位元可用來表示位移量 (`disp`)，無法直接填入 20 位元的記憶體實體位址。組譯器必須計算目標位址與目前執行指標的「相對距離」：

1. **PC 相對定址 (`p = 1, b = 0`)**：
   - 公式：$\text{disp} = TA - (PC)$
   - $(PC)$ 在執行該指令時已指向下一條指令的起始位址。
   - 檢查：$\text{disp}$ 必須介於 $-2048$ 與 $+2047$ 之間。若在此範圍內，組譯器填入此 `disp` 並設定 `p = 1`。
2. **基底相對定址 (`p = 0, b = 1`)**：
   - 若位移量超出 PC 相對定址的範圍，組譯器將嘗試使用基底相對定址。
   - 公式：$\text{disp} = TA - (B)$
   - 組譯器必須透過虛擬指令 `BASE` 得知暫存器 `B` 當前被賦予的數值。
   - 檢查：$\text{disp}$ 必須介於 $0$ 與 $4095$ 之間。若在此範圍內，組譯器設定 `b = 1`。
3. **錯誤處理**：若兩者皆超出範圍，則程式設計師必須改用 Format 4 (即 `+` 指令) 進行絕對定址。

## 2.4 機器獨立的組譯器功能

這些功能不依賴於特定的 CPU 架構，而是為了提升組合語言的靈活性與管理效率。

### 2.4.1 字面值 (Literals) 的處理
字面值允許程式設計師直接在指令中使用常數，而不需要在程式末端用 `BYTE` 或 `WORD` 命名一個變數。例如：`LDA =C'EOF'` 或 `ADD =X'05'`。
- **LITTAB (Literal Table，字面值表)**：當組譯器在原始碼中看到 `=` 開頭的字面值時，會將其記錄在 `LITTAB` 中，記錄其名稱、長度與值。
- **字面值池 (Literal Pool)**：在 Pass 1 結束或遇到虛擬指令 `LTORG` 時，組譯器會集中在記憶體中分派空間給這些字面值常數，並為其分配實體位址。

### 2.4.2 符號定義：`EQU` 與 `ORG`
- **`EQU` (Equate)**：為符號定義常數或表達式。例如：`MAXLEN EQU 4096`。這不會佔用任何記憶體空間，僅在編譯期作為符號替換。
- **`ORG` (Origin)**：重設當前的位置計數器 `LOCCTR`。這通常用於定義複雜的資料結構（如聯集或結構體欄位）。

### 2.4.3 程式區段 (Program Blocks) 與 控制區段 (Control Sections)
- **程式區段 (Program Blocks)**：允許程式設計師在原始碼中自由交錯撰寫程式碼和資料，組譯器會自動在 Pass 1 將它們分類並重新排列，使程式碼集中、資料常數集中，優化記憶體配置。
- **控制區段 (Control Sections)**：將大型程式分割為多個可以獨立編譯與組譯的模組。不同控制區段之間的符號引用必須透過 `EXTDEF`（外部定義）與 `EXTREF`（外部參考）虛擬指令宣告，並由連結器在後續階段進行整合。

## 2.5 單行程組譯器 (One-Pass Assembler) 設計

有些系統為了節省磁碟讀寫時間或記憶體空間，採用**單行程組譯器 (One-Pass Assembler)**。它在一次掃描中就必須產生目的碼。

### 2.5.1 前向參考的解決方案
因為只有掃描一次，當遇到前向參考（如跳躍到尚未定義的標記 `LOOP`）時，組譯器無法得知其位址：
1. **指令部分留白**：組譯器先產生一條目的碼，但將其位址欄位留空。
2. **建立鏈結表 (Linked List of Forward References)**：在 `SYMTAB` 中，將該標記的狀態設為「未定義」，並將此留白指令的位址鏈結到該標記下。
3. **反向回填 (Backpatching)**：當掃描到後面，終於定義了該標記（如 `LOOP` 處）時，組譯器會取得其真實位址，並沿著鏈結表一路往回尋找所有之前留空的指令，生成額外的 T Record (Text Record) 來修改並填入正確的位址，或者直接在記憶體中修改它們。
