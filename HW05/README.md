## 📖 核心概念說明 (Part 1)

### 1. 執行緒 (Thread)
*   **定義**：執行緒是作業系統能夠進行運算排程的最小單位。它被包含在行程（Process）之中，是行程中的實際運作單位。
*   **特性**：同一個行程內的多個執行緒會**共享該行程的記憶體空間**（例如：全域變數、堆區 Heap），但各自擁有獨立的暫存器、程式計數器（Program Counter）與堆疊（Stack）。相比於行程，建立與切換執行緒的開銷非常小，故常被用於高並行的多任務處理。

### 2. 競爭情況 (Race Condition)
*   **定義**：當兩個或多個執行緒同時存取並試圖修改同一個共享資源（變數、檔案等），且最後的結果取決於這些執行緒**執行的順序或交錯時機**時，就會發生競爭情況。
*   **成因**：高階語言的一行指令（如 `counter = counter + 1`），在 CPU 底層會被拆解為多條機器碼指令（讀取、加法、寫回）。在沒有同步機制的情況下，執行緒在執行這些底層指令的途中可能會被 CPU 時鐘中斷切換，導致多個執行緒交錯修改，覆蓋彼此的寫入結果。

### 3. 互斥鎖 (Mutex)
*   **定義**：互斥鎖（Mutual Exclusion Lock）是一種用於多執行緒同步的鎖定機制，確保在任何時間點，**只有一個執行緒能夠進入臨界區段（Critical Section）**去存取共享資源。
*   **運作**：執行緒在進入臨界區段前必須先嘗試取得（Acquire/Lock）互斥鎖，若鎖已被其他執行緒佔用，該執行緒便會被阻塞（Block）掛起，直到鎖被釋放（Release/Unlock）並重新喚醒。

### 4. 死結 (Deadlock)
*   **定義**：死結是指兩個或多個執行緒在執行過程中，因爭奪資源而造成的一種**互相等待、永無止境卡死**的現象。
*   **產生死結的四個必要條件**（ must simultaneously hold ）：
    1.  **互斥 (Mutual Exclusion)**：資源一次只能被一個執行緒使用。
    2.  **持有並等待 (Hold and Wait)**：執行緒已持有至少一個資源，但又申請新的資源，且新資源已被其他執行緒佔用，此時該執行緒在等待時並不釋放自己已持有的資源。
    3.  **不可搶奪 (No Preemption)**：資源不能被強制剝奪，只能由持有它的執行緒主動釋放。
    4.  **循環等待 (Circular Wait)**：存在一個執行緒等待鏈，如執行緒 A 等待執行緒 B 的資源，B 等待 C，C 又等待 A。

## 🛠️ 實作程式說明 (Part 2, 3, 4, 5)

本專案使用 **C 語言 (POSIX Threads)** 實作了三個經典的多執行緒同步問題。

### 1. 銀行存提款模擬 (`bank.c`)
*   **目標**：模擬同一個人存款與提款各 100,000 次，最終存款金額必須精確無誤（初始 1000 元，最後仍為 1000 元）。
*   **解決方案**：
    *   聲明全域變數 `balance`（餘額）為臨界資源。
    *   使用 `pthread_mutex_t lock` 互斥鎖保護 `balance` 的修改區段。
    *   存款線程與提款線程在修改 `balance` 前必須先執行 `pthread_mutex_lock(&lock)`，修改完畢後立刻執行 `pthread_mutex_unlock(&lock)`，保證加減法操作的原子性。

### 2. 生產者消費者問題 (`producer_consumer.c`)
*   **目標**：協調多個生產者與消費者執行緒，確保生產者不會在緩衝區滿時寫入資料，消費者不會在緩衝區空時讀取資料，且緩衝區的索引修改不會發生衝突。
*   **解決方案**：
    *   採用環狀緩衝區（Circular Buffer）作為共享媒介。
    *   使用三個 POSIX 訊號量（Semaphore）：
        1.  `empty`：記錄緩衝區中尚餘的空位數，初始值為 `BUFFER_SIZE` (5)。生產者在寫入前執行 `sem_wait(&empty)`。
        2.  `full`：記錄緩衝區中已被填寫的產品數，初始值為 0。消費者在讀取前執行 `sem_wait(&full)`。
        3.  `mutex`：用作互斥鎖，保護對緩衝區索引 `in` 與 `out` 的並行存取，初始值為 1。
    *   程式設計了總生產數限制（`TOTAL_ITEMS = 20`），生產與消費達到上限後執行緒會安全結束。

### 3. 哲學家用餐問題 (`philosopher.c`)
*   **目標**：5 位哲學家坐在圓桌旁，桌上有 5 支筷子，每人必須同時拿到左右兩側的筷子才能吃飯。若所有人同時拿起左邊筷子，將會導致循環等待，造成死結。
*   **解決方案**：
    *   使用互斥鎖陣列 `pthread_mutex_t forks[5]` 代表 5 支筷子。
    *   **打破循環等待的死結預防策略（不對稱拿取法）**：
        *   **偶數號哲學家 (0, 2, 4)**：先拿左筷子，再拿右筷子。
        *   **奇數號哲學家 (1, 3)**：先拿右筷子，再拿左筷子。
        *   例如：當 0 號和 1 號同時飢餓時，0 號搶奪 0 號筷子（左），1 號搶奪 2 號筷子（右），他們不會同時競爭相同的首選筷子，從而打破了所有人拿同側筷子所形成的環狀等待鏈，徹底避免了死結。
    *   程式設定每位哲學家吃滿 3 餐後退出，模擬可正常且流暢地結束。

## 💻 編譯與執行指引

### 編譯所有程式
請在終端機中切換至 `HW05` 目錄，若環境支援 `make`，可直接執行：
```bash
make
```
若在 Windows/MinGW 環境中不支援 `make`，請使用 `gcc` 手動編譯：
```bash
gcc -Wall -pthread bank.c -o bank
gcc -Wall -pthread producer_consumer.c -o producer_consumer
gcc -Wall -pthread philosopher.c -o philosopher
```

### 執行程式

1.  **執行銀行存提款模擬**：
    ```bash
    ./bank
    ```
    *   **預期輸出**：最後餘額必然為 1000 元，證明 Mutex 成功保護了臨界資源。

2.  **執行生產者消費者模擬**：
    ```bash
    ./producer_consumer
    ```
    *   **預期輸出**：生產者將產品依序放入槽位，消費者依序取出。槽位被交替使用，最終在累計消費達 20 次後安全退出。

3.  **執行哲學家用餐模擬**：
    ```bash
    ./philosopher
    ```
    *   **預期輸出**：各個哲學家自由交替思考與用餐，完全沒有發生死結卡死的情形，且所有人在吃滿 3 次飯後順利離開餐桌。

### 清除編譯檔案
```bash
make clean
```
或手動刪除可執行檔：
```bash
rm -f bank producer_consumer philosopher *.exe
```
